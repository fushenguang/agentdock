# 落地路线：从理念到可执行工程

本页把前两页的结论转成执行路径，并明确哪些是当前事实、哪些是规划目标。

## 当前事实（仓库现状）

- 当前仓库尚未提供独立的 AgentDock CLI 可执行包。
- 已有治理基础能力：OpenSpec 工作流、对齐检查、架构检查、docs 同步。

因此，CLI 路线应从“契约先行”开始，而不是先写大量命令处理代码。

## Phase 0：先定协议，不先堆功能

目标：形成机器可消费且可测试的契约基线。

交付物：

- 命令输入 schema（参数、默认值、约束）。
- 标准输出 schema（success/data/error）。
- 错误码与退出码映射表。

验收标准：

- 同一输入在本地与 CI 产出一致结构。
- 错误分型稳定，不依赖自然语言解析。

## Phase 1：最小可用 CLI（无头优先）

目标：先跑通最短链路。

交付物：

- 最小 init 流程（下载模板、写盘、最小校验、失败回滚）。
- silent + json 输出模式。

验收标准：

- 机器模式 stdout 零噪音。
- 失败路径返回结构化错误对象。

## Phase 2：人类交互与网络鲁棒性

目标：在不破坏无头契约的前提下提升可用性。

交付物：

- TTY 交互流程（仅作为表面层）。
- 代理、镜像、重试、超时、离线策略。

验收标准：

- Human 模式与 Machine 模式共享同一执行器。
- 网络异常可恢复且诊断信息可读。

## Phase 3：Agent 投影（MCP）

目标：把 CLI 能力暴露为 Agent 可直接调用的工具面。

交付物：

- mcp 子命令与工具列表。
- 工具输入输出 schema 与幂等性标注。

验收标准：

- 工具调用参数可由 schema 直接约束。
- 调用失败可通过 code + suggested_action 做自动恢复。

## Phase 4：高阶能力扩展

目标：在稳定协议上扩展价值密度。

交付物：

- add/doctor/migrate 等高频组合命令。
- 模板能力与知识资产（skills/schema/recipes）协同版本发布。

验收标准：

- 高阶命令不破坏既有协议兼容性。
- 文档、skills、行为三者版本一致。

## 度量指标（建议）

- 解析稳定性：结构化输出解析成功率。
- 恢复成功率：可自动恢复错误占比。
- 冷启动性能：常用命令首包时间。
- 变更可追溯性：能力变更与 OpenSpec 记录关联率。
