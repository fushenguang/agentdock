## Context

①②③ 已全部归档。`apps/docs` 是基于 Fumadocs/Next.js 的平台文档站，已有 `content/docs/`（扫描为 `source`）、`llms.txt`/`llms-full.txt` 路由（`fumadocs-core/source` 驱动，覆盖 `content/docs/` 下所有 MDX）、本地 Orama 搜索。`packages/` 已有 `eslint-config`/`tsconfig`。`roadmap.yaml` 有 6 个条目，archive 目录有 3 个已归档 change。

关键约束：`llms.txt` 路由不需改动——只要 `content/docs/` 下多出 MDX，Fumadocs source 自动扫描覆盖。设计核心是**"把 openspec/ 数据投影到 content/docs/"**，而非改 Fumadocs 路由逻辑。

## Goals / Non-Goals

**Goals:**
- `packages/openspec-docs-sync`（TS，零框架依赖，幂等，读 archive + roadmap → 写 MDX）。
- `apps/docs/content/docs/changelog/` 与 `roadmap/`（由同步引擎产出）。
- `apps/docs` 导航入口（roadmap + changelog 两个页面）。
- 本地 Orama 搜索确认可用（Fumadocs 内建，无需额外接入）。
- CI 集成：`docs:sync` 在 docs build 前自动运行。

**Non-Goals:**
- 不实现 Fumadocs MCP Server、Skills 投影、`ai_context` 全套工具链、双语维护、对外 GEO 推广。

## Decisions

### D1. 同步引擎为独立 package，不耦合 Fumadocs
`packages/openspec-docs-sync` 只做文件 I/O：读 YAML/Markdown → 写 MDX。不 import Fumadocs，保持可移植、可单测。Fumadocs 只需看到 `content/docs/` 下的 MDX 文件，不关心来源。
- 备选：在 `apps/docs` 内写 Next.js route 动态读取 openspec。否决——运行时读文件引入复杂性，且产物不幂等、不可 git-track。

### D2. 生成产物进版本管理，标注"自动生成勿手改"
MDX 产物提交进 git（`content/docs/changelog/`、`content/docs/roadmap/`），理由：Fumadocs 在构建时需要文件存在；CI 可静态验证同步已运行。文件头注释 `<!-- AUTO-GENERATED by openspec-docs-sync. Do not edit manually. -->`。
- 备选：gitignore + CI 每次重新生成。否决——需要保证 CI 环境幂等，且丢失了"变更可 git diff 追踪"的好处。

### D3. llms.txt 无需修改
Fumadocs `source` 扫描 `content/docs/**`，`llms.txt`/`llms-full.txt` 路由直接覆盖新 MDX。④ 不动这两个路由文件。

### D4. changelog 提取字段最小化
从 proposal.md 提取：`roadmap-id`（frontmatter）、归档日期（从目录名 `2026-06-01-<id>` 解析）、Why 段、Capabilities 段、Non-goals 段。不做全文投影（避免 llms.txt 过长）。

### D5. CI 集成：docs:sync 作为 docs build 前置步骤
在 `ci-full.yml`（③ 产出）中或单独 `docs-sync.yml` 中，在 `apps/docs` build 前运行 `pnpm docs:sync`，保证 changelog/roadmap MDX 是最新的。

### D6. 本地 Orama 确认
Fumadocs 内建 Orama，`apps/docs` 已有搜索路由（`app/api/search/route.ts`）。确认路由存在且引用 `source`，changelog/roadmap 页进入索引后自动可搜索。不需要额外接入。

## Risks / Trade-offs

- [归档目录命名变化破坏解析] → D4 从目录名 `YYYY-MM-DD-<id>` 解析日期，格式固定（由 openspec CLI 生成），脆弱性低。
- [生成 MDX 的 Fumadocs frontmatter 格式不符] → 同步引擎输出 frontmatter 时遵循 `pageSchema`（`title` 必填，`description` 可选），防止构建失败。
- [手动编辑生成文件] → 文件头注释 + CONTRIBUTING 说明阻止，不做技术强制（MDX linter 过严会误报）。
- [CI docs:sync 与 build 顺序问题] → 作为 npm script `predocs:build` 或显式 CI step 顺序保证。

## Migration Plan

1. 新建 `packages/openspec-docs-sync`，实现核心同步逻辑，注册到 workspace。
2. 运行一次 `pnpm docs:sync`，生成当前 3 个 changelog 条目 + roadmap MDX。
3. 在 `apps/docs` 导航（`meta.json` 或 source config）加入 roadmap/changelog 入口。
4. 本地验证 docs build 成功、两个页面可访问、llms.txt 覆盖新内容。
5. 接入 CI（docs build 前置步骤）。
- 回滚：删除 `packages/openspec-docs-sync`、`content/docs/changelog/`、`content/docs/roadmap/`，还原导航配置。
